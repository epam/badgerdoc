# Renditions and DZI

## What is a Rendition?

Every document uploaded to Badgerdoc must have its renditions. A rendition is simply a conversion from any format to a list of PNG images with the tag "rendition" added.

### Rendition Requirements

Every document must have at least 1 rendition with:
- **Metadata**: Page number stored in metadata field as JSON:
  ```json
  {
    "page": 1
  }
  ```
- **parent_document_id**: Set to the source document's ID
- **Tag**: "rendition"
- **Format**: PNG

### Multi-page Documents

In case a document has more than 1 page (where a page is a vertical split of the document), each page becomes a separate rendition:
- Page 1: metadata `{"page": 1, "width": <pixels>, "height": <pixels>}`, parent_document_id = source document
- Page 2: metadata `{"page": 2, "width": <pixels>, "height": <pixels>}`, parent_document_id = source document  
- Page N: metadata `{"page": N, "width": <pixels>, "height": <pixels>}`, parent_document_id = source document

Each rendition's metadata always includes:
- `page`: the page number
- `width`: the width of the rendered image in pixels
- `height`: the height of the rendered image in pixels

This ensures that every uploaded document, regardless of its original format, has a standardized PNG representation for further processing and viewing.

## How to Get Renditions?

To get all renditions, there are 2 options:

### Option 1: Filter by Tag and Parent Document ID

Use the [documents list endpoint](/swagger/#/Document/documents_list), filtering by tag "rendition" and parent_document_id.

Example query:
```bash
curl -X 'GET' \
  'http://127.0.0.1/badgerdoc/documents/?parent_document_id=1246&tags=rendition' \
  -H 'accept: application/json' \
  -H 'X-CSRFTOKEN: ***'
```

### Option 2: Dedicated Renditions Endpoint

Use the [document renditions endpoint](/swagger/#/Document/document_renditions_list).

Example query:
```bash
curl -X 'GET' \
  'http://127.0.0.1/badgerdoc/document/1246/renditions/' \
  -H 'accept: application/json' \
  -H 'X-CSRFTOKEN: ***'
```

### Differences Between Options

- **Option 1**: Supports pagination, better for large numbers of renditions
- **Option 2**: Returns all renditions in 1 response, might be slow for documents with many pages

## How Renditions Are Generated?

When a supported format is uploaded, for example PDF, Badgerdoc triggers converter workflow: `BadgerdocPNGConvertWorkflow`. Each workflow does next:

1. Check if renditions already exist and *deletes* them
2. For every page (if multipage) creates rendition document with tag `rendition` and metadata `{"page": N}` and `parent_document_id` targeting to original document

## What is DZI

DZI is format Badgerdoc uses to show any document on UI using OpenSeaDragon library. Every document uploaded to Badgerdoc must have DZI. DZI is *always* generated from PNG rendition. In case of multiple page document, DZI will be generated for every page based on page rendition. DZI at least has 2 tags `dzi` and `xml`.

## How to get DZI

Use endpoint [document DZI list](/swagger/#/Document/document_dzi_list) to get list of DZI files. Each page has separate DZI. Example request:
```bash
curl -X 'GET' \
  'http://127.0.0.1/badgerdoc/document/1246/dzi/' \
  -H 'accept: application/json' \
  -H 'X-CSRFTOKEN: ***'
```

## How DZI are generated

DZI generated as soon as PNG "rendition" created in Badgerdoc, using `BadgerdocDZIConvertWorkflow` workflow. First workflows checks if `dzi` already created and removes it with all layers. After generates new DZI and creates tiles. DZI and it's tiles always has parent_document_id targeting to it's rendition. Important! It's not source document, it's rendition. In case of rendition removal, DZI and all tiles will be removed as well.

## What is tiles?

Tiles is exact rectangles (images) generated by layers. Every tile has as document_parent_id it's rendition. For more info, check OpenSeeDragon and DZI documentation.
